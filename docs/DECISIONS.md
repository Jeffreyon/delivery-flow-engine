# Decisions

| Topic | Current choice | Why it exists today | Notes |
|---|---|---|---|
| Product identity | Generic workspace/account/admin scaffold | Matches the current runtime | Phase 0 hardening baseline |
| External delivery backbone | Treat the sibling `logistics-api` repo as the current DEN or BLN provider for deliveries, events, nodes, and handoffs | The PRD expects an external delivery network API, and that backend now exists in the workspace | Delivery Flow Engine should integrate with it before reviving local-first delivery runtime plans |
| Frontend routing | Separate public, auth, user, and admin trees | Clear shell and auth boundaries | No storefront tree |
| Backend module shape | Controller, service, repository split | Keeps raw SQL code organized | Used across all live modules |
| Data access | Shared `query()` helper with raw SQL | Low abstraction overhead | No ORM |
| Auth transport | Session cookie plus bearer fallback | Local-first auth with simple protected-route support | Login still returns `idToken`, and secure cross-origin deployments use `SameSite=None` on the session cookie while local insecure flows stay `Lax` |
| RBAC model | `users.roles` plus `roles.permissions` | Simple admin and self gating | `users.roles` is an array of role ids; no DB FK |
| BLN client abstraction | The local backend must talk to `logistics-api` through an internal client layer | The PRD explicitly calls for a client layer between app routes and the external delivery API | Frontend callers should never see BLN service secrets, tenant API keys, exchanged tenant tokens, or node session tokens |
| Logistics client env contract | `LOGISTICS_API_URL`, `LOGISTICS_API_SERVICE_SECRET`, and optional `LOGISTICS_API_TIMEOUT_MS` | Slice 12 now needs a stable config surface before route facades land | `LOGISTICS_API_SERVICE_SECRET` must match the sibling `logistics-api` `DELIVERY_BACKEND_SERVICE_SECRET`; it is trusted service auth, not tenant verification |
| Local-to-BLN auth bridge | Keep frontend auth local, then let the backend mint membership-scoped BLN node sessions from the stored tenant API key | Tenant users in this app should access tenant data only through an active tenant membership plus a permitted node assignment, and secure handoffs in `logistics-api` are now node-bound | Normal runtime routes use `/api/node-auth/session` through the backend only; `POST /api/tenant-auth/exchange` remains a support path for explicit admin exceptions |
| BLN tenant integration persistence | Persist the durable tenant integration in `bln_tenant_accounts`, keep local memberships in `bln_tenant_memberships`, and store node act-as rights in `bln_node_assignments` while mirroring only ids into `users.preferences.bln` | The bridge now needs a tenant-level credential store plus local membership and assignment state instead of a loose JSON pointer | Stored shape is one tenant integration to many local memberships and many node assignments; reuse it as company workflows broaden |
| Tenant member management surface | Keep tenant membership and node assignment management under `/api/v1/network/users*`, not generic `/api/users` | These records are BLN bridge state, not general profile state, and they need BLN node validation against the selected tenant | Keep the first management surface admin-only until tenant-scoped admin roles are deliberately introduced |
| BLN token storage | Never persist exchanged tenant access tokens or node session tokens locally | The delivery backend can mint short-lived runtime credentials on demand, and storing them would widen risk without adding durable value | The backend stores only the encrypted tenant API key and returns only local facade responses to the frontend |
| Bootstrap binding rule | Admin bootstrap binds the created tenant and first node to one local user immediately | The first bridge is not useful if bootstrap still leaves context unresolved | `bindUserId` is optional and defaults to the current admin |
| Workspace bootstrap boundary | Keep generic signup local-only, and move BLN tenant bootstrap into explicit client onboarding at `POST /api/v1/network/workspaces/bootstrap` | Tenant provisioning belongs to a client or dashboard instance, not to every user account | Invited users can remain unprovisioned until an existing tenant admin assigns membership and node access; `POST /api/v1/network/provision-self` remains a compatibility alias during the transition |
| Tenant join model | Use durable email-matched invitation records accepted by authenticated local users | The first join flow should not depend on email delivery infrastructure or public magic links | Admins create or revoke invitations under `/api/v1/network/invitations*`, and users accept them from the workspace page after local signup or login |
| Cross-system write boundary | Accept that BLN bootstrap plus local user binding is not a distributed transaction | This repo and `logistics-api` write to different databases | Pre-check the local target user first, then document the absence of a cross-system transaction instead of implying one |
| Remote delivery facade | Expose BLN deliveries and delivery events only through local `/api/v1/deliveries*` routes | The frontend should not call the sibling BLN backend directly, and the app already has a backend bridge and local auth context | Keep the local facade thin and truthfully upstream-shaped until a real UI or projection need justifies reshaping |
| Membership-scoped remote-delivery access | Delivery reads and writes require the caller's active tenant membership, permitted node assignment, and a freshly minted node session | Tenant members should see tenant data only within the tenant and node contexts assigned to them, and the sibling BLN backend now supports node-bound runtime auth | Admins do not get a general delivery or handoff read override in this app; the only explicit admin exception remains handoff resolution |
| Idempotent remote writes | Forward `Idempotency-Key` for local delivery-create and delivery-event append routes | The sibling BLN backend already supports it, and frontend callers should not lose that safety just because they go through the local facade | Keep the header pass-through narrow to write routes |
| Remote handoff facade | Expose BLN handoff status, audit, initiate, retry, verify, dispute, and resolve only through local `/api/v1/handoffs*` plus `/api/v1/deliveries/:id/handoff-status` | The frontend should still not call the sibling BLN backend directly, and custody workflows are the next real product differentiator | Keep the local facade thin and upstream-shaped until the app has a real custody UI or projection need |
| Local handoff resolution gate | Keep `POST /api/v1/handoffs/:id/resolve` admin-only in this app | The sibling BLN backend can resolve through tenant access, but this repo still uses admin as the first operator role and should not widen dispute resolution silently | Admin callers still resolve a target BLN tenant through `tenantId` or a stored binding |
| Delivery API versioning | New BLN-backed app routes should start under `/api/v1`, while scaffold routes remain under `/api/*` | Avoids retrofitting the current scaffold contract while preserving a clean local app boundary above the external BLN API | Do not add direct frontend calls to the sibling BLN backend in milestone 1 |
| Milestone 1 BLN actors | `user` plus a BLN tenant membership and node assignment covers normal runtime access, while `admin` keeps bootstrap and support-only exceptions | The current repo only has `admin` and `user` roles, and the first real local binding model now supports multi-user tenant membership without widening local role taxonomy yet | Dedicated `operator` or `driver` local roles remain deferred until the app has a real multi-actor workflow |
| Local delivery schema role | Keep `orders`, `drivers`, `deliveries`, and `assignments` as dormant local extension or projection candidates, not the current source of truth | Those tables exist in schema, but the live external BLN backend already owns deliveries, events, and handoffs | Do not keep expanding local delivery schema until a deliberate projection or augmentation decision is made |
| Response model | Raw JSON success payloads and `{ message }`-style errors | That is what current controllers and error handler emit | No uniform success envelope today |
| User mutation surface | Shared `PUT /api/users/:id` route with actor-scoped field validation | Preserves the existing path while removing self-service access to admin-only fields | Route-level updates should only touch existing users |
| Notification mark-read | Authenticated route with ownership-scoped repository update | Minimal feed support with a safe write boundary | Response is still minimal |
| Event model | Public `GET /api/events` and admin-only `POST /api/events`, plus public `GET /api/delivery-events` and admin-only `POST /api/delivery-events` | Restores platform-wide observability while keeping delivery-specific logs isolated | Delivery writes should create a parent `events` row plus a child `delivery_events` row, and child types should stay delivery-scoped |
| Delivery event schema direction | `events` is the parent event ledger and `delivery_events` is the delivery-specific child ledger | The scaffold still needs generic platform events for base infrastructure behavior, not only logistics logs | Later delivery slices should build lifecycle-owned writes on top of the parent-child event model |
| Tracking and incidents | Local tracking and incident records are now follow-on app features on top of BLN-backed deliveries, not the first execution path | The active gap is still the missing external client layer and local facade routes | Revisit local telemetry or incident schema only after the BLN integration path is working |
| Async platform direction | BullMQ plus Redis are now the shared async foundation | The app can use them later for BLN projection refresh, stalled-handoff alerts, notification fanout, or diagnostics | Queue bootstrap lives under `backend/src/core/queue` and the worker entrypoint is `backend/worker.js` |
| Async worker topology | Start with one Railway `worker` service deployed from the `backend` path | A single worker is simpler than preemptive queue or process splitting | The worker service should use `npm run worker` and stay monolithic until real job load proves otherwise |
| Async platform env vars | `REDIS_URL`, `BULLMQ_PREFIX`, and `WORKER_CONCURRENCY` form the active queue runtime contract | Later job slices need stable env names and worker defaults | Queue names such as `dispatch`, `delivery-status`, `notifications`, `stalled-deliveries`, and `event-reconciliation` can layer on top of this baseline |
| Settings model | One global settings row | Keeps config simple in Phase 0 | Public read only |
| Schema path | `db:migrate` is the primary schema path; `db:init` delegates to it | Aligns local, CI, and preserved SQL behavior | Historical baseline SQL remains append-only |
| Frontend guidance location | `frontend/docs/DESIGN_SYSTEM.md`, `frontend/docs/MOBBIN_DESIGN_WORKFLOW.md`, and `frontend/intelligence.md` | Keeps reusable guidance in repo-true paths | Route inventory still belongs in `docs/UI_ARCHITECTURE.md` |
| Brand baseline | Generic workspace baseline derived from current UI | Matches the current dashboard-heavy template | No retail reference is active |
| Auth bootstrap ownership | Protected route loaders own auth bootstrap | Removes duplicate boot-time `/api/auth/me` requests from a global provider plus loaders | Public routes do not eagerly hydrate auth state today |
| Mobile dashboard nav | User and admin shells use desktop sidebars plus mobile dialog navigation | Preserves the current shell model while covering small screens | Keep the patterns visually aligned across both dashboard trees |

## Gaps and recommended targets
| Current state | Gap | Recommended target |
|---|---|---|
| `PUT /api/users/:id` is actor-scoped instead of broad upsert | Dedicated self and admin routes do not exist | Add dedicated routes only if a real consumer needs clearer boundaries |
| Event create routes are admin-only | Signed or internal-only ingestion is still not modeled for either platform or delivery events | Revisit only when a non-admin producer becomes necessary |
| The repo now has a live sibling BLN backend available | The backend-owned bridge now reaches network context, deliveries, events, and handoffs through membership-scoped node sessions, but the frontend still lacks operator custody surfaces | Use the existing bridge as the only integration seam and move next to UI or projection work instead of adding direct BLN browser calls |
| Protected route loaders own auth bootstrap | Public surfaces do not currently expose auth-aware personalization | Keep bootstrap loader-owned until a real public auth dependency appears |
| Backend lint, typecheck, and build scripts are missing | Validation proof is uneven across layers | Add those scripts when the template runtime is ready to enforce them |
| Mobile dashboard nav exists for the current shells | Some secondary pages still need incremental state and copy cleanup | Tighten touched pages incrementally instead of broad UI rewrites |

## Open questions
- Should user profile writes stay on one route long-term, or split when real edit flows land?
- Should the dormant local `orders`, `drivers`, `deliveries`, and `assignments` schema become a projection cache later, or remain unused until the app adds business objects that the BLN backend does not own?
